DỮ LIỆU
1. Lựa chọn Công nghệ (Tech Stack)
•	Backend: Java and Java Framework.
•	Database chính (RDBMS): PostgreSQL.
o	Vai trò: Lưu trữ dữ liệu chính cần tính chính xác tuyệt đối.
•	Database bổ trợ (NoSQL): MongoDB.
o	Vai trò: Để quản lý Session, lưu trữ Feedback và Log hành vi để làm Dashboard phân tích trải nghiệm.
•	Thanh toán: dùng code và mock test của vnpaysandbox cho việc xử lý toàn bộ luồng và kết quả thanh toán.
•	Gửi mail xác thực: dùng dữ liệu thật với smtp gửi mail với form giao diện thông báo sẽ dùng model giao diện mẫu gửi vào người dùng.
•	Implement a Pessimistic Locking strategy in SQL during booking to prevent double-booking. 
•	Use NoSQL for the ”Hotel Catalog,” prioritizing Availability (AP) for amenity searches.Create a SQL Trigger that notifies a log table whenever a room rate is changed by >50%. 
•	 Use SQL Window Functions to rank room performance within each hotel based on revenue.
•	use Oracle SQL Developer / MS SQL Studio for SQL and MongoDB Compass for NoSQL.
 
2. Phân phối dữ liệu (Data Distribution)
Database	Nội dung quản lý	Lý do chọn
PostgreSQL	Users, Profiles, Branches, Rooms, Bookings, Payments.	Cần tính ACID để tránh mất mát dữ liệu tài chính và lỗi đặt trùng phòng.
MongoDB	Sessions, Mail Tokens, Feedbacks, Activity Logs, Room Cache.	Cần sự linh hoạt, tốc độ ghi nhanh và khả năng lưu trữ dữ liệu dạng Document (JSON) phức tạp.

3. Data Model Chi tiết (Schema Design) chỉ là căn bản có thể mở rộng và nâng cấp như thêm ảnh/lists ảnh vào dữ liệu ở khung database.
A. Nhóm PostgreSQL (Dữ liệu quan hệ)
1.	Users: id (UUID), email, password_hash, role, is_active, created_at.
2.	Profiles: user_id (FK), full_name, phone, avatar_url, address.
3.	Branches: id, name, address, phone, email.
4.	RoomTypes: id, branch_id (FK), name, base_price, capacity, amenities (JSONB).
5.	Rooms: id, room_type_id (FK), room_number, status (Available, Occupied, Maintenance).
6.	Bookings (Orders): id (UUID), customer_id, room_id, check_in, check_out, total_price, status.
7.	BookingServices: id, booking_id (FK), service_id, quantity, actual_price.
 
B. Nhóm MongoDB (Dữ liệu Document)
8.	Collection sessions:
o	Cấu trúc: { _id, user_id, ip, device_info, expires_at }.
o	Đặc điểm: Sử dụng TTL Index để tự động xóa session khi hết hạn.
9.	Collection verification_tokens:
o	Cấu trúc: { token_hash, email, type, expires_at }.
10.	Collection feedbacks:
o	Cấu trúc: { booking_id, user_id, rating (1-5), content, images: [], manager_reply: {}, metadata: {} }.
o	Lý do: Giúp lưu nhiều ảnh và các phản hồi lồng nhau dễ dàng hơn SQL.
11.	Collection activity_logs:
o	Cấu trúc: { user_id, action (SEARCH/VIEW), target_id, timestamp, details: {} }.
o	Lý do: Phục vụ Dashboard thống kê hành vi khách hàng.
4. Mối liên hệ giữa các bảng (Cross-DB Mapping)
Vì dùng hai Database khác nhau, mối liên hệ sẽ không dùng Foreign Key vật lý (Hard Link) mà dùng Logical Link (ID tham chiếu):
•	Khi truy vấn một Booking ở Postgres, Backend sẽ lấy booking_id đó để tìm các Feedbacks tương ứng trong MongoDB.
•	User ID là liên kết xuyên suốt cả hai hệ thống để đồng bộ thông tin.
 
5. Giải pháp cho các vấn đề trọng tâm
5.1. Chống Double Booking (Đặt trùng phòng)
•	Vẫn thực hiện tại PostgreSQL: Đây là quy tắc vàng. Dù dùng MongoDB cho nhiều thứ, nhưng khi đặt phòng, bạn phải dùng Database Transaction của Postgres kèm lệnh SELECT ... FOR UPDATE để khóa bản ghi phòng đó lại cho đến khi thanh toán xong hoặc hết hạn giữ phòng.
5.2. Dashboard biểu đồ trực quan (Admin)
Sự kết hợp này mang lại trải nghiệm Dashboard cực tốt:
•	Doanh thu/Lợi nhuận: Lấy từ PostgreSQL (Query SQL SUM, GROUP BY rất mạnh).
•	Hành vi khách hàng: Lấy từ MongoDB (Aggregation Framework). Vẽ biểu đồ: "Tỉ lệ khách xem phòng nhưng không đặt là bao nhiêu?" dựa trên activity_logs.
5.3. Tìm kiếm & Bộ lọc (Search/Filter)
•	Dữ liệu phòng (tên, giá) nằm ở Postgres, nhưng có thể đưa các trường như average_rating từ MongoDB sang bảng RoomTypes ở Postgres.
•	Mỗi khi có Feedback mới ở Mongo, một tiến trình nền sẽ tính lại điểm trung bình và cập nhật vào Postgres. Việc này giúp tìm kiếm phòng theo "5 sao" chạy cực nhanh mà không cần JOIN giữa 2 DB.
•	Ngoài ra có nhiều chi nhánh nên có thể lọc chi nhánh cho người dùng chọn vị trí họ muốn để đặt.
