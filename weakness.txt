ĐIỂM YẾU CHÍNH CỦA DỰ ÁN (Advanced Database Systems - Hotel Booking)
==============================================================

1. POLYGLOT ACID - CHƯA CÓ DISTRIBUTED TRANSACTION (🔴 CRITICAL)
   - PostgreSQL và MongoDB chạy độc lập, không có saga/outbox/XA
   - Booking commit ở Postgres nhưng feedback lỗi ở MongoDB → inconsistent
   - Thiếu: Saga pattern, Outbox pattern, Event sourcing, Compensation transactions
   
2. PESSIMISTIC LOCKING - KHÔNG TOÀN DIỆN (🔴 CRITICAL)
   - ✓ Booking create: CÓ PESSIMISTIC_WRITE trên room
   - ✗ Booking markPaid: KHÔNG có lock
   - ✗ Booking checkIn/checkOut: KHÔNG có lock
   - ✗ Payment create/update: HOÀN TOÀN KHÔNG có lock
   → Race condition ở payment, markPaid, checkIn/Out

3. IDEMPOTENCY DESIGN - YẾU (🟠 HIGH)
   - ✓ Payment.booking_id: UNIQUE (ngăn 2 payment cho booking)
   - ✗ Payment.transaction_ref: KHÔNG UNIQUE (callback có thể duplicate)
   - ✗ Booking: KHÔNG có idempotency_key
   - ✗ createPayment: Client retry → tạo 2 payment với booking_id khác
   → Dễ tạo duplicate payment nếu network retry

4. TRIGGER/FUNCTION/VIEW - CÓ NHƯNG CHƯA KHAI THÁC (🟠 HIGH)
   - ✓ Có: fn_hold_room_booking, fn_confirm_room_booking (WITH lock + idempotency)
   - ✗ NHƯNG: BookingService.java không gọi stored procedure
   - ✗ Vẫn dùng JPA service code thay vì procedure tối ưu
   - ✓ Có view cho reporting
   - ✗ NHƯNG: KHÔNG có view để detect conflict, overlap

5. CONFLICT RESOLUTION - CƠ BẢN, KHÔNG HÌNH THỨC (🟠 HIGH)
   - DB: Partial unique index (70% chặn overlap)
   - App: existsActiveOverlap() check (gap race condition)
   - Lock: PESSIMISTIC_WRITE (chỉ booking)
   - ✗ THIẾU: Exclusion constraint (chặn date overlap 100%)
   - ✗ THIẾU: Formal conflict detection view

6. TEST SUITE - CHỈ HAPPY PATH (🟠 HIGH)
   - ✓ BookingPaymentRoomFlowL1Test: Sequential flow ✓
   - ✓ AuthAndSecurityFlowL1Test: Auth flow ✓
   - ✗ KHÔNG: Concurrency test (2 booking cùng room)
   - ✗ KHÔNG: Duplicate callback test (VNPay callback 2 lần)
   - ✗ KHÔNG: Rollback scenario
   - ✗ KHÔNG: Cross-store consistency test (Postgres + Mongo)

7. ERROR HANDLING - THIẾU RETRY LOGIC (🟡 MEDIUM)
   - ✗ KHÔNG có @Retry cho OptimisticLockingFailureException
   - ✗ KHÔNG có retry cho DeadlockLoserDataAccessException
   - ✗ KHÔNG có idempotent handling cho DataIntegrityViolationException
   - ✗ KHÔNG có circuit breaker cho VNPay integration

8. DATABASE DESIGN - CHƯA DÙNG ADVANCED FEATURES (🟡 MEDIUM)
   - ✗ KHÔNG có: Exclusion constraint (DATE range overlap prevention)
   - ✗ KHÔNG có: Deferrable foreign key
   - ✗ KHÔNG có: Generated columns (tính duration)
   - ✓ Có: Partial indexes (tốt)
   - ✓ Có: Constraints & FK (tốt)

9. API DESIGN - THIẾU VERSIONING (⚪ LOW)
   - ✗ KHÔNG có: API versioning (/v1, /v2)
   - ✗ Nếu thêm conflict resolution → API contract thay đổi
   - ✗ Breaking change cho client

10. MONITORING - KHÔNG CÓ LOCK METRICS (⚪ LOW)
    - ✗ KHÔNG track: Lock wait time, deadlock count, abort rate
    - ✗ KHÔNG track: Booking overlap detection rate
    - ✗ KHÔNG có: Observable metrics cho ACID/concurrency

11. FRONTEND - CÓ UI NHƯNG CHƯA THẬT SỰ MẠNH (🟠 HIGH)
   - ✗ Dùng rất nhiều inline style trong layout/page/component → khó maintain, khó scale design system
   - ✗ UI chủ yếu dựa vào class chung + style rải rác, chưa có kiến trúc theme/token rõ ràng
   - ✗ `PublicLayout`, `DataTable`, `RoomList`, `PreviewBooking` vẫn còn nhiều logic trình bày trong component
   - ✗ Phần lớn filter/sort/pagination xử lý ở client → dễ lệch với dữ liệu thật khi backend đổi state
   - ✗ `RoomList` fetch toàn bộ rooms rồi mới filter/sort/paginate, chưa có server-side pagination/search cho dữ liệu lớn
   - ✗ Booking flow phụ thuộc nhiều vào `location.state` và client state, nếu refresh hoặc đi trực tiếp URL thì dễ mất ngữ cảnh
   - ✗ `authStore` dựa nhiều vào localStorage/sessionStorage + timer refresh token → còn yếu về độ bền session và bảo mật thực tế
   - ✗ Không thấy chiến lược optimistic UI / rollback UI rõ cho booking/payment retry, nên trải nghiệm khi lỗi mạng chưa tốt
   - ✗ Error/empty/loading states có nhưng còn cơ bản, chưa có skeleton/placeholder phong phú cho mọi màn hình quan trọng
   - ✗ Routing chia theo role khá nhiều nhưng nhiều trang vẫn là form/list đơn giản, chưa tạo cảm giác demo "nâng cao" về UX
   - ✗ Component common có tái sử dụng nhưng chưa đủ chuẩn design system nghiêm túc: typography, spacing, motion, responsive behavior còn rời rạc
   - ✗ Frontend chưa thể hiện rõ câu chuyện "polyglot ACID" hay conflict handling bằng UI feedback mạnh (conflict banner, retry, merge hint, lock notice)

12. FRONTEND - BỐ CỤC, CTA VÀ WORDING CÒN RỜI RẠC (🟠 HIGH)

TÓNG KẾT THEO MỤC TIÊU MÔN HỌC
=================================================================================

Yêu cầu môn "Hệ Cơ Sở Dữ Liệu Nâng Cao":
  ✓ Polyglot ACID (PostgreSQL + MongoDB) → CÓ setup, NHƯNG CHƯA dùng distributed transaction
  ✓ Pessimistic Locking cho booking/payment → CÓ ở booking, NHƯNG CHƯA ở payment
  ✓ Trigger/Function/View giải quyết conflict → CÓ nhưng CHƯA được khai thác
  ✓ Chuẩn dự án sinh viên → ✓ Đạt (code structure, modularity, security)

⚠️ KẾT LUẬN:
Dự án phù hợp cho "Hệ Cơ Sở Dữ Liệu CĂNG BẢN" nhưng chưa đủ cho "NÂNG CAO"
Cần 1-2 tuần làm việc để bổ sung:
  - Distributed transaction (Saga/Outbox)
  - Toàn diện Pessimistic Locking (payment, transitions)
  - Exclusion constraints + conflict detection view
  - Comprehensive concurrency testing
  - Formal idempotency & compensation pattern
   - Frontend state flow, UX và error feedback mạnh hơn để demo đúng chất "nâng cao"
